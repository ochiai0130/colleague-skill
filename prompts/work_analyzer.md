# Work Skill 分析プロンプト

## タスク

**{name}** の原材料（ドキュメント、メッセージ、メールなど）を受け取ります。
そこから仕事の能力と手法を抽出し、Work Skill の構築に使用します。

**原則：仕事に関連する内容のみ抽出し、雑談は無視する。推測はせず、根拠がある場合のみ記載し、なければ「原材料不足」と注記する。**

---

## 共通抽出ディメンション（全職種共通）

### 1. 担当範囲

原材料から識別する：
- 担当しているシステム/モジュール/事業ライン/プロダクト
- メンテナンスしているドキュメント（APIドキュメント、wiki、runbook...）
- 職責の境界（何が自分の範囲で、何が範囲外か）
- 頻繁に言及するプロジェクトコード名、業務用語

```
出力形式：
担当領域：[記述]
コアシステム：[リスト]
メンテナンスドキュメント：[リスト]
境界：[担当範囲/担当外]
```

### 2. 業務フロー

タスク記述、会議議事録から抽出する：
- タスクを受けた際の処理手順
- 方針/ドキュメントを書く際の構成習慣
- 進捗管理とデッドライン対応の方法
- 異常/緊急事態の対応方法

```
出力形式：
タスク受領：[手順]
方針策定：[構成の記述]
異常対応：[フロー]
```

### 3. 出力形式の好み

- 表/リスト/フローチャート/プレーンテキストの使い分け
- 結論先行か順序立てて説明するか
- ドキュメントの詳細度（極めて簡潔/適度/詳細）
- 返信/メールのスタイル

```
出力形式：
ドキュメントスタイル：[記述]
詳細度：[極めて簡潔/適度/詳細]
```

### 4. 経験ナレッジベース

本人が明確に述べた経験的判断、踏んだ地雷、技術的見解（原文を直接引用）：

```
- "[原文または要約]"
- "[原文または要約]"
```

## 職種別専門抽出

{name} の職種に応じて、対応するディメンションを重点的に抽出する：

---

### 🖥️ バックエンドエンジニア / サーバーサイドエンジニア

**技術規範**：
- 技術スタック（言語、フレームワーク、ミドルウェア）
- 命名規則（APIパスのスタイル、変数/関数の命名）
- API設計（レスポンス構造、エラーコード、ページネーション、冪等性）
- データベース操作の好み（ORM vs 生 SQL、トランザクション境界）
- 例外処理スタイル

**Code Review の重点**：
- 繰り返し指摘する CR の問題（N+1、トランザクション、並行安全性...）
- CR コメントのスタイル（直接的/婉曲、[block]/[suggest] のレベル分け...）

**デプロイと運用**：
- 注目している監視指標
- 本番障害の調査手順
- 変更リリースフロー

---

### 🌐 フロントエンドエンジニア

**技術規範**：
- 技術スタック（フレームワーク、状態管理、スタイリング方式）
- コンポーネント分割の原則（いつ分割し、いつ分割しないか）
- パフォーマンスの関心事（初期表示、遅延読み込み、bundle サイズ...）
- API呼び出しとエラーハンドリングの方式

**エンジニアリングプラクティス**：
- コード規約ツール（ESLint ルール、Prettier 設定の好み）
- テストカバレッジ要件（ユニットテスト/統合テストへの態度）
- CR の重点（アクセシビリティ/レスポンシブ/互換性への関心度）

---

### 🤖 アルゴリズムエンジニア / ML エンジニア

**研究と実験**：
- 問題定義の方法（ML問題の分解方法）
- 実験設計の習慣（ベースライン選択、ablation 設計）
- 指標定義の好み（オフライン指標 vs オンライン指標への態度）
- よく使うモデル/方法論

**エンジニアリング実装**：
- 学習フレームワークの好み
- モデルデプロイフロー
- データ処理規範

**ドキュメントと結論**：
- 実験レポートの書き方（結論重視/過程重視）
- 引用する論文や方法論

---

### 📱 プロダクトマネージャー / テクニカルプロダクトマネージャー

**要件処理**：
- PRD の構成と詳細度
- ユーザーストーリー/要件境界の定義方法
- 開発チームとのすり合わせ方法（レビュー方式、修正フロー）

**意思決定フレームワーク**：
- 優先順位の決め方（RICE/MoSCoW/独自方式）
- データドリブン vs 直感の比率
- 要件の衝突への対処方法

**成果物**：
- 納品するドキュメントの種類（PRD/MRD/プロトタイプ/競合分析）
- プロトタイプツールの好み
- データトラッキングへの関与度

---

### 🎨 デザイナー

**デザイン規範**：
- 使用するデザインシステム/コンポーネントライブラリ
- 注記方法と納品規範
- pixel-perfect への要求度

**業務フロー**：
- 要件からデザイン案までのステップ
- レビュー/検収の方法
- 開発側の再現度の問題への対処方法

---

### 📊 データアナリスト

**分析手法**：
- よく使う分析フレームワーク（ファネル/コホート/A/B テスト...）
- SQL のスタイル（簡潔/コメント詳細）
- データ可視化の好み（チャートの種類選択）

**レポートスタイル**：
- 結論 vs データの比率
- 「データで語る」の実行度
- データ異常/定義の齟齬への対処方法

---

## 出力要件

- 言語：日本語
- 情報がないディメンション：`（原材料不足、関連ドキュメントの追加を推奨）` と注記する
- 原文の根拠がある結論：引用符で原文を注記する
- 出力結果は work.md の生成に直接使用されるため、具体的で実行可能な記述にし、「かもしれない」「傾向がある」などの曖昧な表現は避ける
